# [보고서 12] API, Supabase DB 연동 및 추천 알고리즘 설계 분석

본 문서는 **TasteMate**의 현재 백엔드(FastAPI)에서 API 통신, Supabase 데이터베이스 연동, 그리고 맞춤형 추천 알고리즘이 어떻게 구현되고 맞물려 동작하는지를 기술합니다.

---

## 🔗 1. Supabase DB 연동 구조 (Database Architecture)

현재 TasteMate 백엔드는 Supabase가 제공하는 PostgreSQL 데이터베이스와 REST API 방식을 통해 통신합니다.

### A. 연결 방식 (`supabase_client.py`)
- 백엔드에서는 공식 Python SDK인 `supabase-py`를 사용합니다.
- `.env`에 저장된 `SUPABASE_URL`과 **`SUPABASE_SERVICE_ROLE_KEY`** 수준의 키를 사용하여, 백엔드 서버가 DB의 모든 테이블에 권한 제약 없이 직접 접근(Admin 권한)하여 데이터를 읽고 씁니다.

### B. 주요 활용 테이블
1. **`profiles` (사용자 프로필)**:
   - 사용자의 신체 정보, 목표 체중 관리 방식 등에 따라 **일일 목표 칼로리(target_calories)** 등을 저장합니다. 추천 알고리즘 쿼리 시 이 지표를 조회합니다.
2. **`meals` (식사 기록)**:
   - 사용자가 당일 섭취한 음식의 칼로리 총합을 계산하기 위해 조회됩니다. (목표 칼로리에서 남은 여유 칼로리를 계산하는 데 쓰임)
3. **`recommendation_history` (추천 내역)**:
   - 사용자에게 어떤 식당/메뉴가 왜 추천되었는지를 기록하여, 추후 알고리즘 학습(또는 MLOps) 시 활용할 수 있는 기반 데이터를 쌓습니다.

---

## 🧠 2. 하이브리드 추천 알고리즘 설계 (Recommendation Algorithm)

추천 로직은 `recommend.py` 엔드포인트에 구현되어 있으며, **LLM(초거대 언어 모델)의 추론 능력**과 **외부 검색 API(네이버 로컬)**를 결합하여 설계된 하이브리드 RAG (Retrieval-Augmented Generation) 패턴의 초기 형태입니다.

### A. 컨텍스트 수집 (데이터 어그리게이션)
사용자가 추천을 요청하면 백엔드는 5가지 컨텍스트를 즉시 수집합니다.
1. **User Profile**: `profiles` 테이블에서 목표와 기준 타겟 칼로리를 가져옵니다.
2. **Recent Meals**: `meals` 테이블에서 오늘 먹은 식사 총합을 계산하여 **남은 섭취 가능 칼로리**를 산출합니다.
3. **Location (Reverse Geocoding)**: 카카오 로컬 API를 통해 사용자의 좌표(lat, lng)를 "서울시 강남구 역삼동" 형태의 실제 동네 주소 문자열로 치환합니다.
4. **Time & Weather**: 요청된 시간대와 현재 날씨 정보를 입력받습니다.

### B. LLM 기반 동적 쿼리 생성 (`o3-mini`)
기존에는 단순히 "맛집"을 검색했다면, 현재는 수집된 컨텍스트를 OpenAI 모델(`o3-mini`)에 프롬프트로 전달하여 상황에 가장 **최적화된 검색 키워드**와 **짧은 어드바이스**를 동적으로 생성합니다.
- **예시 프롬프트 결과**: 목표가 다이어트이고 남은 칼로리가 적으며 비가 오는 경우 -> 검색어: `"고단백 저칼로리 국물"`, 조언: `"비 오는 날 따뜻하지만 가볍게 즐겨보세요!"`

### C. 외부 API 기반 식당 도출 (네이버 로컬 API)
- LLM이 만들어준 똑똑한 검색어(`query`) 앞에 사용자의 현재 위치 문자열(`location`)을 붙여(예: `역삼동 고단백 샐러드`) **네이버 로컬 API**를 호출합니다.
- 네이버 API에서 실제 평점이 높거나 코멘트가 많은 순서로 상위 5개의 식당 데이터를 받아옵니다.

### D. 데이터 정제 및 최종 응답
- 네이버 데이터의 부족한 부분(음식 사진, 상세 영양성분 등)을 자체 Default 로직으로 보강합니다.
- 각 추천 항목에 LLM이 작성해준 **"추천 사유(reason)"**를 동적으로 주입합니다.
- 최종 정제된 리스트를 프론트엔드로 반환하는 동시에 `recommendation_history` DB에 **비동기적으로 로그를 적재**합니다.

## 🚀 3. 알고리즘 설계의 한계 및 고도화 방향
1. **네이버 API 의존성**: 현재 카테고리나 영양 성분이 백엔드 자체 식당 DB(Foods DB)가 아닌 외부 의존 및 Default Fallback에 묶여있어, 정밀한 탄단지(매크로) 계산에는 한계가 있습니다.
2. **고도화 계획**: Part 3 이후, 실제 `foods_v2` (음식 영양 정보 데이터) 테이블과 Vector DB를 연계하는 본격적인 RAG 랭킹 모델(`ranker.py`)로 교체하여 더 수학적이고 정교하게 재정렬(Re-ranking)하는 방식을 도입해야 합니다.
