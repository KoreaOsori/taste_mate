## 1. 프로젝트 성과 요약 (Executive Summary)

### 🚀 핵심 성과
본 프로젝트는 현대인의 가장 큰 고민인 "오늘 뭐 먹지?"를 AI와 풍부한 데이터를 바탕으로 해결하기 위해 기획되었습니다. 초기 기획의 광범위한 범위를 핵심 MVP(Minimum Viable Product) 중심으로 재편하여, **추천-지도-기록**으로 이어지는 완결성 있는 사용자 경험을 구축하였습니다. 단순한 UI 구현을 넘어, 실제 메뉴 데이터셋 구축부터 클라우드 배포까지의 Full-Stack 개발 전 과정을 성공적으로 완수하였습니다.

### 🏆 주요 구현 성과
-   **데이터 중심 추천**: 식약처 영양 데이터를 정제하여 1끼 기준 메뉴 메타 데이터셋(`foods_v2`) 구축 및 RAG(검색 증강 생성) 기반 추천 엔진 구현.
-   **사용자 중심 UX**: 틴더 스타일의 스와이프 인터페이스와 카카오맵 기반의 직관적인 동적 지도 연동, 음식점의 네이버 주소 연결, 선택한 음식 캘린더와 연동하여 당일 영양 데이터 기록 및 반영.
-   **개인화 데이터 수집 및 추천 적용**: 개인의 취향 선호도 이외에도 위치, 날씨, 기분 등 실시간 개인화 데이터를 수집하여 상황 맞춤형 추천 쿼리 생성.
-   **하이브리드 추천 스코어링 엔진**: 사용자의 영양정보, 현재 날씨, 기분, 동행인, 예산 등 5~6가지 변수를 복합적으로 연산하여 상위 후보를 추출하는 정교한 랭킹 시스템 구축.
-   **강력한 폴백(Fallback) 시스템**: 네트워크나 외부 API 연결 실패를 대비하여 카테고리별 기본 이미지 및 위치 미제공 시의 대체 검색 정책 등 예외 처리 구현.
-   **서비스 완성도**: Vercel & Railway를 통한 클라우드 배포로 외부 접근이 가능한 프로덕션 급 환경 구축.

## 2. 기술적 구현 상세 (Technical Implementation)

### 2.1 시스템 아키텍처 및 스택 (Docs 00, 03 참조)
-   **Frontend**: React (TypeScript), Vite, Tailwind CSS, Framer Motion.
-   **Backend**: FastAPI (Python), Playwright (Headless Scraping), Uvicorn.
-   **Database**: Supabase (PostgreSQL) - 사용자 프로필, 식사 기록, 추천 이력 관리.
-   **Third-party APIs**: OpenAI (GPT-4o-mini), Kakao Map/Login, Naver Local/Image Search.

### 2.2 추천 알고리즘 및 엔진 상세 (Docs 13, 17 참조)
1.  **1차 랭킹 (Heuristic Scoring)**: `foods_v2` 데이터셋을 기반으로 잔여 칼로리, 날씨, 기분, 예산 등 수학적 가중치를 계산하여 상위 후보군 필터링.
2.  **2차 추론 (LLM Reasoning)**: GPT-4o-mini가 후보군 중 사용자의 상황에 가장 적합한 메뉴를 최종 선정하고, 정량적인 추천 사유와 최적의 검색 쿼리를 생성.
3.  **3차 실시간 컨텍스트 결합 (Location & Context)**: 
    - 사용자의 현재 위치(행정동)와 선정된 메뉴를 결합하여 **네이버 플레이스** 실시간 검색 수행.
    - **나노바나나(스크레이핑)** 엔진을 통해 검색된 식당의 실제 영업 정보와 전화번호를 추출하여 최종 사용자에게 전달.
4.  **데이터 영속성**: 모든 추천 이력과 사용자의 관심도(좋아요/싫어요)를 DB에 기록하여 개인화의 기반 마련.

## 3. 트러블슈팅 및 리스크 관리 (Challenges & Solutions)

프로젝트 개발 과정에서 발생한 주요 기술적 난제들과 그 해결 과정을 상세히 기록합니다.

### 3.1 [데이터] 식품 영양 DB와 실제 메뉴 단위의 불일치
-   **현상**: 공공 데이터셋의 메뉴 단위가 실생활(1끼 식사)과는 거리가 멀어 추천 품질이 낮아짐.
-   **원인**: 기성 DB는 식재료나 가공식품 중심의 영양 성분 나열에 치중했기 때문.
-   **해결**: 식약처 데이터를 원천으로 하되, '비빔밥', '김치찌개' 등 사용자가 인지하는 1끼 메뉴 단위로 데이터를 재구성하고 추천용 메타데이터(맛, 상황, 어울리는 날씨)를 직접 설계 및 태깅함.

### 3.2 [위치] 위치 미제공 시의 부적절한 검색 결과 (강남 쏠림 현상)
-   **현상**: 사용자가 위치를 제공하지 않을 때 기본값인 '강남' 맛집만 보여주는 정보 비대칭 문제 발생.
-   **원인**: 이전 로직에서 위치 부재 시 특정 지역(강남)을 고정 검색어로 사용했기 때문.
-   **해결**: `use_location` 상태값을 세분화하여 위치 미제공 시에는 지역 키워드 없이 **"메뉴명 맛집"**으로만 검색하게 하여 보편적인 맛집 리스트를 제공하도록 개선함.

### 3.3 [UX] 지도 렌더링 지연 및 API 초기화 오류
-   **현상**: 상세 정보 팝업을 열 때 지도가 로딩되지 않거나 백지 화면이 노출됨.
-   **원인**: React의 컴포넌트 생명주기와 카카오맵 SDK의 로딩 시점 불일치.
-   **해결**: `useEffect`와 `MapInitializer` 컴포넌트를 분리하여 SDK가 완전히 로드된 후 지도를 동적으로 초기화하도록 로직을 강화하고, 실패 시 '다시 시도' 버튼을 제공함.

### 3.4 [데이터 수집] 실시간 영업 시간 추출의 어려움
-   **현상**: 표준 검색 API에서 영업시간 정보를 제공하지 않아 사용자가 헛걸음할 리스크 존재.
-   **원인**: 포털 사이트의 정책상 상세 영업 정보는 API가 아닌 웹 페이지에서만 제공됨.
-   **해결**: **Playwright(Headless Browser)**를 백엔드에 통합하여 네이버 플레이스 상세 페이지를 실시간 스크레이핑하는 "나노바나나" 엔진 구현.

### 3.5 [배포] 클라우드 환경의 포트 매핑 및 의존성 충돌
-   **현상**: Railway 클라우드 배포 후 백엔드가 시작되지 않거나 스크레이핑 모듈이 작동하지 않음.
-   **원인**: 
    1. Railway는 동적 포트(`$PORT`)를 사용하는데 고정 포트(8000)로 설정되어 연동 실패.
    2. Docker 환경에 브라우저 구동을 위한 리눅스 시스템 라이브러리 부재.
-   **해결**: 
    -   `entrypoint.py`에서 `$PORT` 환경 변수를 최우선적으로 할당받도록 수정.
    -   `Dockerfile`을 최적화하여 Chromium 및 관련 시스템 의존성을 레이어에 포함시켜 해결.

### 3.6 [UX] 페이지 전환 시 추천 상태 증발 현상
-   **현상**: 추천 결과를 보고 있다가 탭을 옮기거나 새로고침을 하면 스틱 카드가 사라지는 문제.
-   **원인**: 프론트엔드의 React State가 새로고침 시 초기화되기 때문.
-   **해결**: **`sessionStorage`**를 연동하여 추천받은 카드 리스트와 질문 단계 상태를 브라우저 세션 동안 보존하도록 고도화함.

## 4. 자체 평가 및 향후 계획

### 📊 자체 평가
-   제한된 자원 속에서도 **데이터 설계부터 배포까지의 풀사이클(Full-Cycle)**을 경험하며 MVP를 넘어선 완성도를 달성함.
-   문제 발생 시 단순히 기능을 축소하지 않고, 스크레이핑이나 동적 환경 변수 처리 등 기술적 대안을 찾아 해결함.

### 🚀 향후 로드맵
1.  **모바일 최적화 (PWA & Native)**: 
    - 현재의 웹 기반 프론트엔드를 **PWA(Progressive Web App)**로 전환하여 앱처럼 설치 가능한 사용자 경험 제공.
    - 장기적으로 **React Native/Expo**를 도입하여 iOS/Android 네이티브 앱 출시.
2.  **개인화 고도화**: 사용자 관심도 데이터를 기반으로 한 머신러닝 기반 개인화 추천 강화.
3.  **커뮤니티 확장**: 추천받은 식당의 실시간 리뷰 공유 및 동행 구인 기능.

## 4. 트러블슈팅 및 리스크 관리 (Challenges & Solutions)

### 4.1 [데이터] 식품 영양 DB와 실제 메뉴의 괴리 (Docs 16 참조)
-   **문제**: 원천 데이터가 식재료 중심이어서 '1끼 식사' 단위의 추천이 어려움.
-   **해결**: 메뉴 데이터셋을 재정의하고 추천에 필요한 메타데이터(맛, 상황 등) 컬럼을 직접 설계 및 정제.

### 4.2 [기술] 클라우드 환경의 기술적 제약 (Docs 20 참조)
-   **문제**: Railway 배포 시 포트 설정(`$PORT`) 및 브라우저 의존성 문제로 인한 빌드 실패.
-   **해결**: `entrypoint.py`에서 동적 포트를 할당받도록 수정하고, `Dockerfile`에 Playwright 라이브러리 의존성을 포함시켜 해결.

### 4.3 [UX] 날짜 불일치 및 상태 유지 (Docs 18 참조)
-   **문제**: UTC 기준 처리로 인한 날짜 밀림 현상 및 새로고침 시 데이터 소멸.
-   **해결**: 로컬 타임존 기반 처리로 수정하고, `sessionStorage`를 도입하여 새로고침 시에도 추천 상태 보존.

## 5. 자체 평가 및 향후 계획

### 📊 자체 평가
-   제한된 인원과 자원 속에서도 **데이터 설계부터 배포까지의 풀사이클(Full-Cycle)**을 경험하며 MVP를 넘어선 고도화된 기능을 구현함.
-   단순한 기획에 그치지 않고 기술적 한계(API 데이터 부족 등)를 스크레이핑 등의 기술로 정면 돌파함.

2.  **커뮤니티 연동**: 식단 공유를 넘어선 소셜 기능 및 사용자 간 피드백 시스템.
3.  **데이터 확장**: 현재 정제된 1,000여 개의 메뉴 데이터셋을 만 개 단위로 확장 및 품질 관리.
