# 추천 로직 상세 문서 및 위치 정책

이 문서는 Taste Mate **실시간 맛집 추천** 파이프라인 전체와 **위치 미제공 시 정책**을 정리합니다.

---

## 1. 전체 흐름 요약

```
[프론트] 질문 플로우 (메인/디저트, 바로 vs 상황 맞춤)
    → GET /api/v1/recommend/{user_id}?lat=&lng=&weather=&hour=&emotion=&companion=&preference=&budget=&use_location=
[백엔드] 프로필·오늘 식사 조회 → foods_v2 후보 랭킹 → LLM 검색어·조언 생성
    → (실제 위치 있음) 행정동 + 메뉴명으로 네이버 맛집 검색
    → (위치 없음) 메뉴명만으로 “비슷한 맛집” 검색 (특정 지역 거짓 표기 없음)
[프론트] 카드 스와이프, 좋아요/싫어요 → POST /recommend/interest, 식사 기록 연동
```

---

## 2. 위치 정책 (중요)

- **위치가 있을 때**  
  - 사용자가 위치 허용을 했고, 실제 좌표(`lat`, `lng`)가 들어오면  
  - Kakao로 행정동(동/구)을 구한 뒤, **"{행정동} {메뉴명} 맛집"** 형태로 네이버 로컬 검색  
  - → **진짜 주변 맛집**을 추천합니다.

- **위치가 없을 때**  
  - 위치 허용을 하지 않았거나, 좌표를 보내지 않는 경우  
  - **과거처럼 “강남”을 기본값으로 쓰지 않습니다.**  
  - 대신 **`use_location=false`** 로 호출하고, 백엔드에서는 **지역 키워드 없이**  
  - **"{메뉴명} 맛집"** 만으로 네이버 검색합니다.  
  - → “근처”가 아니라 **메뉴/취향에 맞는 비슷한 맛집**을 보여주고,  
  - 특정 지역(예: 강남)을 거짓으로 전제하지 않습니다.

- **API 동작**  
  - 쿼리 파라미터 **`use_location`**  
    - `true`: 위도·경도 기반 주변 검색 (프론트는 실제 위치 있을 때만 `true` 전달)  
    - `false`: 위치 미사용 → 메뉴 기반 “비슷한 맛집”만 검색, 강남 등 기본값 사용 안 함
    - 생략: `lat`/`lng`가 있으면 주변 검색, 없으면 비슷한 맛집만 검색

---

## 3. 백엔드 파이프라인 상세

### 3.1 입력

| 파라미터   | 설명 |
|-----------|------|
| `user_id` | 사용자 ID (프로필·식사·이력 조회용) |
| `lat`, `lng` | 위도·경도 (선택). 없으면 주변 검색 안 함 |
| `use_location` | `true`/`false`. `false`면 지역 키워드 미사용, “비슷한 맛집”만 검색 |
| `weather`, `hour` | 날씨·시간 (랭커·LLM용) |
| `emotion`, `companion`, `preference`, `budget` | 질문 플로우에서 온 값 (메인/디저트 분기 포함) |

### 3.2 단계별 처리

1. **프로필·오늘 식사**  
   - Supabase `profiles` → 목표 칼로리 등  
   - 오늘 식사 합산 → `consumed_calories`

2. **디저트 플로우 여부**  
   - `preference in ['커피, 음료, 차', '빵, 케이크, 과자']` 이면 디저트 전용 검색어·조언 생성

3. **후보 랭킹**  
   - `foods_v2`(is_active=true) 전부 조회  
   - `RecommendationRanker`로 스코어: 칼로리 적합도, 취향/기분/동행/예산/날씨 매칭  
   - 상위 10개 → LLM에 전달

4. **LLM (GPT-4o-mini)**  
   - 메인: 선정 메뉴 1~3개 + 네이버 검색용 쿼리 + 60자 내 조언  
   - 디저트: 선택 유형에 맞는 검색어 + 조언

5. **맛집 검색 (위치 정책 적용)**  
   - **`use_location == true` 이고** `lat`/`lng`가 있으면:  
     - Kakao 좌표→주소 → 행정동 추출 → `"{행정동} {메뉴명} 맛집"` 검색  
   - **`use_location == false`** (또는 위치 없음):  
     - **지역 키워드 없이** `"{메뉴명} 맛집"` 만 검색 → 비슷한 맛집만 반환, 기본값(강남) 사용 안 함

6. **이미지**  
   - 네이버 이미지 검색 → 실패 시 카테고리별 기본 이미지

7. **이력**  
   - `recommendation_history`에 추천 결과 저장  
   - 스와이프 좋아요/싫어요 → `POST /recommend/interest` 로 `selected` 갱신

---

## 4. 프론트엔드 연동

- **화면**: `RestaurantRecommendationScreenNew.tsx`  
- **위치 사용 여부**  
  - `userProfile.location_consent === true` 이고 실제 `userLocation`(좌표)이 있을 때만  
  - `use_location: true` 로 요청  
  - 그 외(위치 비허용, 또는 아직 좌표 미수신)에는 **`use_location: false`** 로 요청  
- 이렇게 해서 “위치 없을 때는 강남 기본값이 뜨면 안 되고, 근처가 아닌 비슷한 맛집만 보여준다”는 요구사항을 만족합니다.

## 6. 환경 변수

- `OPENAI_API_KEY`: LLM 검색어·조언 생성  
- `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`: 네이버 로컬/이미지 검색  
- `KAKAO_REST_API_KEY` 또는 `KAKAO_NATIVE_APP_KEY`: 좌표→주소 (위치 사용 시)  
- `GOOGLE_PLACES_API_KEY`: (선택) Google Places API 키. 설정 시 **실제 식당 이미지**를 Find Place → Place Details → Place Photo로 조회하고, 실패 시 네이버 이미지 검색으로 폴백. 키는 HTTP referrer 제한 권장.
