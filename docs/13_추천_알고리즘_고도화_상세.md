# [보고서 13] 추천 알고리즘 고도화(RAG 및 LLM 랭킹) 상세 분석

본 문서는 TasteMate 프로젝트의 핵심 기능인 **사용자 맞춤형 추천 알고리즘**이 어떻게 고도화되었는지 상세하게 다룹니다. 기존 네이버 로컬 API와 단순 프롬프트에 의지하던 방식에서 벗어나, 자체 음식 메타데이터(`foods_v2`)와 추론형 LLM(`gpt-4o-mini`)을 결합한 하이브리드 파이프라인으로 개선되었습니다.

---

## 🚀 1. 고도화 파이프라인 개요 (Data Flow)

새로운 추천 시스템은 크게 4단계의 프로세스를 거칩니다.

1. **사용자 컨텍스트 수집 (Frontend -> Backend)**
   - 프로필 정보: 사용자 기본 정보 (이름, 체중, 감량 목표 등)
   - 식사 기록(`meals`): 오늘 섭취한 칼로리를 기반으로 **남은 일일 권장 칼로리(잔여 칼로리)** 계산
   - 실시간 환경: 위치(GPS), 날씨, 현재 시간대
   - **[NEW] 사용자 인앱 응답 4종**: 프론트엔드 추천 화면에서 선택한 감정(`emotion`), 동행자(`companion`), 음식 취향(`preference`), 예산(`budget`)

2. **후보군 필터링 및 1차 스코어링 (Ranker - What to Eat)**
   - Supabase의 `foods_v2` 테이블에서 활동 중인(`is_active=True`) 모든 음식 메타데이터를 가져옵니다.
   - `models/ranker.py`의 `RecommendationRanker`가 사용자의 현재 컨텍스트를 기반으로 각 음식들을 다각도로 수학적 스코어링합니다. (가중치 부여)
   - 가장 점수가 높은 상위 N개(현재 10개)의 음식 후보(`Top Candidates`)를 추려냅니다.

3. **고도화된 LLM 추론 및 검색어 생성 (GPT-4o-mini - Why to Eat)**
   - 상위 10개의 음식 후보와 사용자의 구체적인 프로필/상황 정보를 `gpt-4o-mini` 모델에 프롬프트로 전달합니다.
   - LLM은 이 중에서 사용자의 상황에 가장 완벽하게 부합하는 1~3개의 메뉴를 최종 선택합니다.
   - 동시에, 왜 이 메뉴가 추천되었는지에 대한 **"친절하고 정량적인 사유(Reason)"**와, 식당 검색을 위한 **"최적의 네이버 API 검색어(Query)"**를 JSON 형태로 생성합니다.

4. **실제 로컬 식당 매핑 (Naver API - Where to Eat)**
   - LLM이 반환한 검색 쿼리를 네이버 로컬 API에 던져 주변의 실제 식당 정보를 수집합니다.
   - 검색된 식당 정보와 LLM이 선택한 음식(`selected_foods`)의 영양 정보(탄수화물, 단백질, 지방, 칼로리 등)를 매핑하여 `RestaurantRecommendation` DTO를 조립합니다.
   - 최종 결과 프론트엔드 반환 및 `recommendation_history` 테이블에 이력 저장 (비동기 로깅).

## 🧠 2. 추천 엔진 스코어링 로직 분석 (`ranker.py`)

로컬 랭커(`RecommendationRanker`)는 `foods_v2`의 풍부한 메타데이터와 사용자의 요청 파라미터를 하나씩 대조하여 점수(`score`)를 누적합니다.

### 2.1. 영양 매칭 (Nutrition Match)
- 사용자의 오늘의 남은 칼로리와 음식의 기본 제공 칼로리의 오차를 계산합니다.
- `오차 = |남은 칼로리 - 음식 칼로리|`
- 칼로리 격차가 적을수록 높은 기본 점수(최대 10점)를 부여하여 다이어트/영양 목표가 깨지지 않게 방어합니다.

### 2.2. 사용자 응답 매칭 (User Q&A Match)
- **취향 (Preference)**: '가벼운/든든한/매콤한/담백한' 등의 요청을 `foods_v2`의 `meal_heaviness`, `taste`, `spicy_level` 컬럼과 대조하여 최대 5점의 가산점 부여.
- **감정 (Emotion)**: '피곤한/우울한/기분 좋은' 등의 상태를 음식의 `emotion_state`(예: '위로/안정/회복/축하')와 매핑하여 가산점 부여 (3점).
- **동행인 (Companion)**: '혼자/친구/가족'을 `preferred_people`과 비교하여 점수 부여 (3점).
- **예산 (Budget)**: 선택된 예산(저렴/보통/고급)과 음식의 `price_per_serving` 구간을 대조 (4점).

### 2.3. 환경 컨텍스트 매칭 (Context Match)
- 날씨(맑음, 비, 눈)를 `foods_v2`의 `preferred_weather`와 대조하여 감성적인 매칭 점수를 부여 (2점).
- 마지막으로 동점자 방지 및 추천 다양성 확보를 위해 미세한 랜덤 소수점을 추가하여 최종 리스트를 정렬(`sort`)합니다.

## 🤖 3. GPT-4o-mini 추론 모델 (LLM Prompting)

기존 단순 키워드 추출용이던 `o3-mini` 대신, 복잡한 상황 인지와 다중 후보군 비교 추론이 가능한 `gpt-4o-mini`를 도입했습니다.

### 주요 프롬프트 전략
- **정량적 피드백 지시**: LLM에게 추천 사유를 작성할 때 단순히 "맛있기 때문입니다"가 아닌, *"남은 칼로리 500kcal에 맞춰 단백질 30g이 포함된 메뉴입니다"*처럼 수치(Nutrition Macro)를 명시하여 신뢰도를 높이도록 엄격하게 지시합니다.
- **후보군 제한 (Constrained Generation)**: 백지에서 환각(Hallucination)으로 없는 메뉴를 만들어내는 것을 방지하기 위해, `ranker.py`가 1차 검증한 10개의 **데이터베이스 내 메뉴 이름 리스트**를 제공하고 무조건 이 안에서만 선택하도록 강제합니다. (RAG 방식의 일환)
- **JSON Structured Output**: 검색어(`query`), 추천사유(`advice`), 채택된 후보음식 배열(`selected_food_names`)을 정확한 JSON 포맷으로 강제 반환받아 백엔드 파싱 오류를 최소화합니다.

---

## 📈 4. 향후 로드맵 연계점
이 고도화된 파이프라인은 추후 AI 알고리즘의 확장성을 크게 높여줍니다.
1. `foods_v2` 테이블의 데이터(음식 수)가 아무리 많아져도, `ranker.py`를 통해 1차적으로 LLM이 처리할 데이터를 안전하게 축소할 수 있습니다.
2. 현재 Heuristic(휴리스틱 규칙) 기반의 `Ranker`를 미래에는 실제 유저 클릭/만족도 데이터를 바탕으로 한 LightGBM 또는 Vector DB (Cosine Similarity) 모델로 손쉽게 교체할 수 있는 구조적 분리가 완료되었습니다.
