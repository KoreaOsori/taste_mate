# TasteMate UI/UX 개선 가이드

> 현재 프론트 구현을 바탕으로, 어떤 순서로 UI/UX를 고민하면 좋을지 정리한 문서입니다.

---

## 1. 우선 해결할 구조적 이슈 (접근성)

지금 **화면은 있는데 들어갈 수 없는** 부분이 있습니다. UX보다 먼저 “기능에 도달할 수 있게” 만드는 게 좋습니다.

| 이슈 | 현재 상태 | 제안 |
|------|-----------|------|
| **식사 기록(meal-log)** | 홈/네비에 링크 없음 | 홈 상단 또는 칼로리 카드 옆에 “오늘 식사 기록하기” 버튼 추가 → `onNavigate('meal-log')`. 또는 캘린더의 “식사 추가”와 동일 플로우로 통합 검토. |
| **AI 챗봇(chat)** | 홈/네비에 링크 없음 | 홈 “오늘의 식습관 조언” 카드에 “AI에게 물어보기” 버튼 추가, 또는 플로팅 버튼/하단 네비 한 칸으로 진입 제공. |
| **푸드팜(foodfarm)** | 어디서도 진입 불가 | 건강식품 구매가 서비스 범위라면: 프로필 메뉴 또는 홈 카드에서 “건강식품 보기” 링크 추가. 범위가 아니면 화면 제거 또는 나중 기능으로 보류. |

**고민 포인트**

- “식사 기록”을 **홈에서 강조**할지, **캘린더만**으로 할지 정해야 합니다. 홈이 “오늘 한눈에 보기”라면 홈에서 식사 추가 진입이 자연스럽습니다.
- **챗**은 자주 쓰는 기능이면 네비에 두고, 가끔 쓰면 홈/조언 카드에서만 들어가게 하는 식으로 우선순위를 정하면 됩니다.

---

## 2. 정보 구조(IA)와 네비게이션

- **하단 네비**: 홈 / 캘린더 / 추천 / 커뮤니티 / 프로필 (5개, 추천이 중앙 강조).
- **챗·식사 기록·푸드팜**은 네비에 없고, 서브 화면으로만 존재.

**UI/UX 고민 질문**

1. **핵심 행동이 뭔가?**  
   “오늘 먹은 것 기록” vs “맛집 고르기” vs “AI 상담” 중에서, 사용자가 매일 하는 행동을 1~2개 정해 그걸 가장 쉽게 누를 수 있게 두는 게 좋습니다.
2. **탭 수 유지 vs 확장**  
   탭을 5개 이상으로 늘리면 한 손 조작 시 답답해질 수 있음. “식사 기록”을 캘린더와 합치거나, “더보기” 메뉴로 챗/푸드팜을 묶는 방식도 고려.
3. **진입 경로 일관성**  
   “식사 추가”가 캘린더에서는 다이얼로그, meal-log에서는 전체 화면인 점을 통일할지(둘 다 같은 추가 폼 컴포넌트 사용 등) 정하면 유지보수와 사용자 학습에 유리합니다.

## 3. 홈(대시보드) 화면

- 인사말, 위치, 커뮤니티/챌린지 캐러셀, **오늘의 식습관 조언**, **칼로리·3대 영양소 진행률**, **식습관 팁** 캐러셀으로 구성.

**UI/UX 고민 질문**

1. **첫 화면에서 “다음 행동”이 보이는가?**  
   지금은 정보가 많고 “지금 뭘 하면 되는지”는 상대적으로 덜 드러남.  
   - 예: “오늘 식사 기록하기” CTA를 칼로리 카드 위/아래에 두거나,  
   - “목표까지 300kcal 남음 – 식사 추가” 같은 문구로 유도.
2. **칼로리/영양소가 “오늘” 기준인가?**  
   `todaysMeals`가 비어 있으면 0으로 나옴. 앱 열 때 오늘 날짜로 `mealService.getMeals` 한 번 호출해 채워두면, 홈 숫자가 실제와 맞습니다.
3. **캐러셀 개수**  
   커뮤니티·팁 캐러셀이 여러 개면 스와이프 부담이 커질 수 있음. “한 번에 하나만 강조” 또는 “리스트로 한 화면에 2~3개” 정도로 단순화 검토.
4. **접근성**  
   카드 제목/숫자에 충분한 대비와 터치 영역(최소 44px) 확보. “오늘의 조언” 같은 핵심 문구는 폰트 크기·줄 간격으로 읽기 쉽게.

## 4. 온보딩·설정

- **온보딩**: 단계가 많고(이름/신체/식사시간/선호/목표/활동량/위치동의), 한 번에 하나씩 묻는 구조는 좋음.  
- **프로필 편집**: 모달로 이름·나이·키·몸무게·식사시간 등 수정.

**UI/UX 고민 질문**

1. **진행 표시**  
   단계가 5단계 이상이면 “2/7” 같은 진행 인디케이터가 있으면 이탈을 줄이는 데 도움됨.
2. **건너뛰기**  
   선호 카테고리·기피 음식 등은 “나중에 설정”으로 건너뛰게 하면 진입 장벽이 낮아짐.
3. **목표 칼로리 노출**  
   Harris-Benedict 결과를 온보딩 마지막에 “추천 목표: 2,100kcal”처럼 보여주고 수정 가능하게 하면 신뢰감이 올라감.
4. **위치**  
   “맛집 추천을 위해 필요해요”처럼 이유를 짧게 쓰고, 거부해도 앱 사용은 가능하다는 문구가 있으면 좋음.

---

## 5. 맛집 추천·캘린더·커뮤니티

- **추천**: 카드 스와이프, 로딩 화면, 피드백 모달, 외부 링크(네이버/배민/요기요).  
- **캘린더**: 월별 보기, 일별 식사·칼로리, 식사 추가 다이얼로그, 건강 리포트 진입.  
- **커뮤니티**: 글 목록, 필터, 글쓰기. Supabase Function 연동.

**UI/UX 고민 질문**

1. **추천**  
   - “마음에 안 들어요” 피드백 시 다음 추천으로 바로 넘어가는지, 이유 선택(가격/맛/거리 등)을 받을지.  
   - 카드가 많을 때 “오늘 추천 3곳”처럼 개수 제한을 두면 부담이 줄어듦.
2. **캘린더**  
   - 날짜별 색/아이콘으로 “목표 달성일” vs “과다/미달”을 한눈에 보이게 할지.  
   - 식사 추가를 “캘린더에서만” 할지, “홈 + 캘린더 둘 다” 할지 정한 뒤 플로우를 맞추는 게 좋음.
3. **커뮤니티**  
   - 글이 없을 때 “첫 글을 올려보세요” + 글쓰기 버튼이 명확한지.  
   - 에러/연동 실패 시 “잠시 후 다시 시도” 메시지와 재시도 버튼이 있으면 좋음.

---

## 6. 일관성·디자인 시스템

- **색**  
   메인은 초록(green-600 등). CTA·강조는 이 색으로 통일해 두면 “클릭 가능한 것”이 보기 쉬움.
- **타이포**  
   제목/숫자/본문 계층(예: 2xl bold / xl / base)을 화면마다 비슷하게 쓰면 일관된 느낌.
- **여백·카드**  
   `rounded-3xl`, `p-6` 등 반복되는 카드 스타일을 공통 컴포넌트(예: `DashboardCard`)로 빼두면 나중에 테마/다크모드 적용이 수월함.
- **에러·로딩**  
   API 실패 시 “다시 시도” 버튼, 로딩 시 스켈레톤/스피너를 화면별로 비슷한 패턴으로 두는 것을 권장.

## 7. 단기 체크리스트 (우선 적용해 볼 만한 것)

1. [ ] 홈에 **“오늘 식사 기록하기”** 버튼 추가 후 `meal-log` 연결.
2. [ ] 홈 **“오늘의 식습관 조언”** 카드에 **“AI에게 물어보기”** 버튼 추가 후 `chat` 연결.
3. [ ] 앱 최초 로드 또는 홈 진입 시 **오늘 날짜 식사** `mealService.getMeals(userId, today)` 호출해 `todaysMeals` 반영.
4. [ ] **푸드팜**을 쓸 계획이면 프로필/홈에서 진입 경로 추가, 아니면 숨기기 또는 제거.
5. [ ] 온보딩에 **진행 단계 표시**(예: 2/7)와, 선택 항목에 **건너뛰기** 옵션 검토.
6. [ ] 추천/채팅 등 **로딩이 긴 화면**에 로딩 UI와 **타임아웃/재시도** 처리 확인.

## 8. 중장기에서 고민할 것

- **알림**: “점심 시간이에요, 식사 기록해 보세요” 등 (PWA/브라우저 알림 또는 나중에 앱 푸시).
- **오프라인/느린 네트워크**: 식사 기록 로컬 저장 후 동기화.
- **다크 모드**: Tailwind 다크 클래스 + `next-themes` 등으로 전환.
- **접근성**: 포커스 순서, aria-label, 키보드만으로도 핵심 플로우 가능한지.
- **성과 지표**: “이번 주 목표 달성일 4/7”처럼 주간 요약을 홈/캘린더에 노출해 동기 부여.
